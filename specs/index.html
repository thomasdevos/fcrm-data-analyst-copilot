Executive Summary

The Challenge

ING's Order Management as a Platform (OMaaP) serves 6-7 countries with centralized payment processing. Currently 5-7 operational support engineers and a few developers daily handle high volumes of repetitive technical, functional, and operational queries from tenant countries, leading to response delays and resource strain that is unsustainable with the projected 5x growth by July 2026.

The Solution

Deploy a multi-agent AI system to provide instant, context-aware responses to tenant queries, leveraging API-based data sources (Inquiry API, Forge, Orange Sharing, SharePoint, ad hoc sources) with authenticated access and strict security standards including ISO/IEC 25010 compliance.

Expected Impact

  • 50-70% reduction in support tickets requiring human intervention
  • <2 seconds AI response time for auto-resolvable queries (50-60% of all queries)
  • 24/7 availability for tenant queries
  • Free up 5-7 FTEs for strategic work

Proposal Overview

This proposal outlines a phased approach starting with a 2-week Discovery & Assessment to evaluate the optimal AI platform (Microsoft 365 Copilot vs Google Gemini), followed by a staged MVP development across three releases, ensuring incremental value delivery and risk mitigation.

Architecture Components

ü§ñ

OMaaP Service Agent

Entry Point

Role: Primary interface between users and the multi-agent system

  • Receives natural language queries from users
  • Determines if it has necessary access and information
  • Routes queries to appropriate specialized agents via Agent Discovery
  • Orchestrates agent-to-agent (A2A) communication
  • Aggregates responses and forms coherent answers
üîç

Agent Discovery

Orchestration

Role: Identifies and routes to appropriate specialized agents

  • Maintains registry of available agents and their capabilities
  • Analyzes query requirements and matches to agent expertise
  • Handles dynamic agent registration and lifecycle
  • Enables intelligent routing based on query type and data requirements

A2A (Agent-to-Agent) Communication

Role: Enables collaboration between agents

  • Enables complex queries requiring multiple data sources
  • Manages context sharing and response aggregation

Data Integration

Role: Standardized protocol for connecting LLMs to data sources

Backend

  • Indexes content from all data sources for fast retrieval
  • Enables semantic search across documentation and knowledge bases
  • Supports RAG (Retrieval Augmented Generation) workflows
  • Provides unified search interface for all agents

Possible Challenges & Mitigation Strategies

High Risk

Data Source Connectivity Limitations

Challenge

MS 365 Copilot cannot access Elastic/Cassandra in ING environment, and creating custom MCP servers is restricted. Gemini integration may face network/firewall constraints.

Impact

Core use case (payment data queries) may be blocked or require significant workarounds, delaying MVP delivery.

Mitigation

  • Discovery focus: Week 1 priority is validating Elastic connectivity with both platforms
  • Fallback option: If M365 blocked, recommend Gemini (already approved on Vista platform)
  • Alternative architecture: API gateway or middleware layer to proxy Elastic queries if direct access restricted
  • Escalation path: If both platforms blocked, present data to ING leadership for policy exception or infrastructure changes
Medium Risk

Multi-Tenancy & Data Isolation Complexity

Challenge

Ensuring strict tenant data isolation (Italy cannot see Germany data) requires robust RBAC implementation in AI agent, with potential for accidental data leakage.

Impact

GDPR violation, loss of trust, potential regulatory penalties if data leakage occurs.

Mitigation

  • Architecture principle: Filter data at query level (not agent level) - inject tenant ID into all Elastic/Cassandra queries
  • Testing rigor: Dedicated security testing sprint with cross-tenant access attempts
  • Leveraging existing auth: Integrate with Harmonia's existing RBAC (don't reinvent)
  • Audit logging: Log all queries with user identity for forensic analysis
Medium Risk

AI Accuracy & Hallucination Management

Challenge

LLMs may generate plausible-sounding but incorrect answers (hallucinations), especially for complex payment scenarios or edge cases not well-represented in training data.

Impact

Incorrect information provided to tenants could cause operational issues, loss of trust, or financial errors.

Mitigation

  • RAG architecture: Ground all responses in retrieved data (Elastic, docs) - minimize generative content
  • Confidence scoring: Agent indicates confidence level and provides source citations
  • Fallback to human: "Escalate to support" option prominently displayed
  • Continuous monitoring: Track thumbs-down feedback and escalations to identify failure patterns
  • Feedback loop: Monthly review of failure cases to improve prompts and knowledge base
Medium Risk

Knowledge Base Currency & Maintenance

Challenge

OMaaP platform evolves continuously (new features, documentation updates). AI agent knowledge base may become stale if not regularly updated.

Impact

Agent provides outdated information, reducing accuracy and user trust over time.

Mitigation

  • Automated sync: Daily/weekly jobs to re-index Forge, Orange Sharing, SharePoint
  • Version tagging: Knowledge base entries tagged with OMaaP version for deprecation
  • Ownership model: Assign responsibility to OMaaP documentation team for AI knowledge base maintenance
  • Staleness detection: Alert when source documents haven't been updated in >90 days
Low Risk

User Adoption & Change Management

Challenge

Tenants and support engineers may be resistant to using AI assistant, preferring familiar human support channels.

Impact

Low usage rates, failure to achieve expected ticket reduction and ROI.

Mitigation

  • User-centric design: Involve support engineers and tenants in POC testing and feedback
  • Clear value prop: Demonstrate speed and convenience advantages in training
  • Hybrid approach: AI complements (not replaces) human support - easy escalation path
  • Gamification: Track and celebrate usage milestones (e.g., "1000 queries resolved!")
  • Training & documentation: Comprehensive onboarding materials for all personas
Low Risk

Performance & Scalability

Challenge

High query volumes from multiple countries may cause latency or API rate limit issues with LLM provider.

Impact

Slow response times degrade user experience, potentially higher costs with pay-per-token model.

Mitigation

  • Caching layer: Cache common queries/responses to reduce LLM API calls
  • Load testing: Stress test during POC to validate performance under expected load
  • Rate limiting: Implement fair usage policies per tenant/user
  • Horizontal scaling: Design for stateless agents that can scale with demand
  • Monitoring: Real-time performance dashboards with alerting on SLA breaches

Risk Management Approach

All identified risks will be tracked in a Risk Register maintained throughout the project, with weekly reviews during MVP sprints. Each risk includes:

  • Probability: Likelihood of occurrence (High/Medium/Low)
  • Impact: Severity if realized (High/Medium/Low)
  • Owner: Team member responsible for monitoring and mitigation
  • Status: Open, Mitigated, Realized, Closed
  • Mitigation Plan: Specific actions and timelines
